# 蓝心九问知识库

更新时间：2026-05-25

用途：本文件整理自蓝心九问官方文档站，作为后续创建 Bot、配置工具库、挂载知识库、调试、发布 API、排查鉴权和错误码时的本地速查资料。官方文档仍可能更新；遇到接口路径、限流、隐私条款或发布规则不确定时，应回到文末链接复核。

## 1. 平台定位

蓝心九问是 vivo 面向 AI 应用和智能体的一站式开发平台，用于构建高质量、可视化、可运营的 Bot。

平台核心能力：

- Bot 开发：创建自主规划型 Bot 或流程编排型 Bot。
- 工具挂载：支持插件、工作流、知识库、数据库等能力。
- 工具库：支持通过 HTTP、IDE、MCP 三种方式创建插件，也可使用平台内置插件。
- 工作流：组合插件、大语言模型、代码块等节点，构建稳定流程。
- 知识库：支持文档型、问答型、表格型知识库，支持上传文件和网页爬取。
- 发布出口：支持 URL、API、嵌入式弹窗等发布方式。
- 运营分析：提供智能体运行报表、对话数据、可追溯调试和运营分析能力。

## 2. Bot 类型

蓝心九问的 Bot 分为两类。

### 自主规划型 Bot

由模型自主判断如何调用工具并执行任务，适合无固定流程、任务形态灵活的场景。

适用例子：

- 通用问答助手。
- 业务咨询助手。
- 需要模型根据用户问题自动选择知识库、插件或工作流的场景。

### 流程编排型 Bot

通过画布编排固定执行流程，适合定向任务和业务专属流程。

适用例子：

- 旅行规划。
- 报告分析。
- 数据处理。
- 对结果准确性、步骤顺序、输出格式有严格要求的任务。

## 3. Bot 创建

在 Bot 模块右上角点击“创建 Bot”，可以创建自己的专属 Bot。

创建配置项包括：

- 头像。
- 名称。
- 描述。
- Bot 类型。

建议：

- 名称写清业务场景，例如“智能日程助手”“财务票据问答 Bot”。
- 描述写清服务对象、能力边界和主要输入输出。
- 如果任务流程明确，优先考虑流程编排型；如果用户问题开放、路径多变，优先考虑自主规划型。

## 4. Bot 调试

完成 Bot 配置后，可在调试窗口直接与 Bot 对话，检查配置是否有效。

调试重点：

- 回答是否符合预期。
- 是否正确调用知识库、插件或工作流。
- 多轮对话是否能保持必要上下文。
- 输出格式是否稳定。
- 不满意当前对话时，可使用“重新开始”清空对话内容。

建议调试用例：

- 正常问题：验证核心能力。
- 边界问题：验证拒答、澄清和兜底。
- 多轮追问：验证上下文。
- 缺少资料的问题：验证是否会编造。
- 需要工具调用的问题：验证工具是否被正确触发。

## 5. 工具库

### 插件

平台支持通过三种方式创建插件：

- HTTP 插件。
- IDE 插件。
- MCP 插件。

平台也集成了若干内置插件，例如新闻热点、网页搜索、内容生产等，可用于拓展 Bot 能力。

### 工作流

工作流可以组合：

- 插件。
- 大语言模型。
- 代码块。
- 其他流程节点。

适合步骤较多、可拆解、可验证的任务。与完全交给模型自由规划相比，工作流更利于稳定结果、控制格式和复用业务流程。

### 知识库

知识库类型：

- 文档型。
- 问答型。
- 表格型。

导入方式：

- 上传文件。
- 网页爬取。

知识库通过向量搜索匹配内容，让 Bot 更准确地回答用户问题。

## 6. 知识库调用

知识库有两类常见调用方式。

自主规划型 Bot：

- 在规则配置页面定位到“知识”。
- 点击“添加”。
- 选择需要关联到 Bot 的知识库。
- 支持添加多个知识库。
- 由模型自动判断何时检索和调用知识库。

流程编排型 Bot 或工作流：

- 在流程编排画布点击底部功能栏“添加节点”。
- 选择“知识库”节点。
- 添加节点后选择知识库类型。
- 文档型/问答型与表格型知识库的检索和输出格式不同，需要分开添加。
- 如果同时使用多类知识库，应添加多个知识库节点分别检索。

实践建议：

- 通用 FAQ、制度文档、产品说明适合文档型或问答型知识库。
- 结构化数据、清单、价格、表格资料适合表格型知识库。
- 对关键业务答案，优先使用知识库而不是只依赖模型记忆。
- 需要确定性流程时，把知识库作为工作流节点显式编排。

## 7. API 发布

蓝心九问支持把创建好的 Bot 发布为可调用 API。通过 HTTP 请求，外部应用、服务或工作流可以和 Bot 交互。

适用场景：

- 智能客服：嵌入网站或应用，提供自动化服务。
- 内容创作：自动生成报告、文案、邮件草稿。
- 数据分析：调用 Bot 对数据做分析和洞察。
- 任务自动化：执行信息查询、数据整理等重复工作。

发布步骤：

1. 登录九问平台。
2. 进入 `Bots` 页面，选择自己创建的 Bot。
3. 点击页面上方的“发布”。
4. 在发布页面选择 `API`。
5. 点击“发布”。

只有发布为 API 渠道后，才能通过 API 查看 Bot 基本设置、发起智能体对话等。

## 8. API 限制

官方文档当前列出的限制：

| 维度 | 限制 |
| --- | --- |
| 关键 API QPS，包括发起对话、执行工作流等 | 20 QPS |
| 上传文件 API QPS | 10 QPS |
| 涉及模型处理的 API 并发 | 大部分模型默认 2 并发 |
| 工作流相关 API 请求体 | 20 MB |
| 其他类别 API 请求体 | 15 MB |

开发建议：

- 调用侧加重试、退避和限流。
- 文件上传与对话请求分开限流。
- 不要把大文件或超长上下文直接塞进普通 API 请求。
- 高并发业务需要提前评估模型并发限制。

## 9. API 基础概念

会话：

- Bot 和用户之间的一段问答交互。
- 一个会话包含一条或多条消息。
- 会话能够自动处理截断，以适应模型上下文长度。

消息：

- 用户或 Bot 创建的一条内容。
- 内容可包含文本、图片或文件。
- 消息以列表形式存储在对话中。

对话：

- 在会话中对 Bot 的一次调用。
- Bot 收到请求后，会结合用户输入和预设配置调用模型或工具执行任务。
- 可直接发起一次对话，也可先创建会话和消息，再在指定会话中发起对话。
- 指定会话中的其他消息会作为历史消息传递给大模型。

上下文段落 Section：

- Bot 对话管理中的独立上下文段落。
- 用于分隔不同对话阶段或主题。
- 创建会话时会生成一个 Section。
- 用户清除上下文时，系统会创建新的 Section，避免新对话受旧历史影响。

## 10. 鉴权

蓝心九问 API 采用 API Key 鉴权。所有发往 API 接口的请求都需要携带密钥，用于验证身份和权限。

获取和使用步骤：

1. 在平台 API 页面进入“API 密钥”。
2. 创建密钥。
3. 妥善保存密钥，泄露可能导致 API 被非法调用。
4. 调用接口时，在请求 Header 中加入鉴权信息。

Header 格式：

```http
Authorization: Bearer {{API-SECRET-KEY}}
```

curl 示例：

```bash
curl --location --request POST 'https://<api-host>/<path>' \
  --header 'Authorization: Bearer {{API-SECRET-KEY}}'
```

注意：

- 文档的参数表中有位置写作 `Authorization: {{API-SECRET-KEY}}`，示例使用 `Authorization: Bearer {{API-SECRET-KEY}}`；实际开发优先按示例的 Bearer 格式实现，并以官方最新文档为准。
- API Key 不要提交到仓库。
- 本地开发建议放在 `.env` 或系统密钥管理中。
- 服务端调用时不要把密钥暴露给前端。

## 11. Bot 接口：查看配置

接口用途：获取指定 Bot 的详细配置信息，例如名称、描述、头像，便于外部应用动态展示 Bot 信息。

当前文档信息：

| 项 | 值 |
| --- | --- |
| 请求路径 | `/parameters` |
| 请求方法 | `GET` |
| 请求域名 | `http://jiuwen-api.vmic.xyz/v1` |
| 备用/内网域名 | `http://jiuwen-api.vivo.lan:8080/v1` |
| Header | `Authorization: Bearer {{API-SECRET-KEY}}` |

示例：

```bash
curl --location --request GET 'http://jiuwen-api.vmic.xyz/v1/parameters' \
  --header 'Authorization: Bearer {{API-SECRET-KEY}}'
```

注意：文档页面的示例里也出现过测试域名，实际业务接入前需要以平台 API 发布页给出的正式地址为准。

## 12. 错误码

官方文档说明：接口错误或异常时，错误码体现在 HTTP code 中，不在返回结果中；返回结果会包含错误描述。

当前列出的 HTTP 错误码：

| HTTP code | 描述 |
| --- | --- |
| 400 | 服务器内部异常、参数错误等 |
| 403 | Bot 大模型限流 |

排查顺序：

1. 是否携带 `Authorization` Header。
2. Header 是否使用 Bearer 格式。
3. API Key 是否正确、有效、未泄露或禁用。
4. Bot 是否已经发布为 API。
5. 请求路径、域名、方法是否与 API 发布页一致。
6. 请求体大小是否超过限制。
7. 是否触发 QPS 或模型并发限制。
8. 参数格式是否符合接口要求。

## 13. 合规与安全提示

隐私政策和服务协议是独立文档，开发和发布前需要单独阅读。知识库与 Bot 可能涉及用户输入、上传文件、图片、文本、语音等数据，接入时应注意：

- 避免上传无授权的个人信息、敏感商业资料或受版权保护内容。
- 对外部用户输入进行最小化收集。
- 不在日志中记录 API Key、个人敏感信息和完整隐私材料。
- 对 Bot 输出加业务侧审核和兜底，尤其是医疗、法律、金融等高风险场景。
- 明确告知用户 AI 生成内容可能存在不准确，需要人工核验。

## 14. 本地项目协作建议

如果后续要在当前快应用项目中接入蓝心九问：

- 前端快应用不应直接保存蓝心九问 API Key。
- 建议通过自有后端代理调用蓝心九问 API。
- 快应用端只请求自有后端，避免密钥泄露。
- 对用户输入和 Bot 输出做长度限制。
- 对 403 限流做友好提示，例如“当前请求较多，请稍后再试”。
- 可将快应用本地业务文档整理为蓝心九问知识库，让 Bot 处理问答。
